Master Test Plan

  1. Introduction

Purpose

The purpose of the Iteration Test Plan is to gather all of the information necessary to plan and control the test effort for a given iteration. It describes the approach to testing the software, and is the top-level plan generated and used by managers to direct the test effort.

This Test Plan for the CLOUD supports the following objectives:

•        Identifies the items that should be targeted by the tests.

•        Identifies the motivation for and ideas behind the test areas to be covered.

•        Outlines the testing approach that will be used.

•        Identifies the required resources and provides an estimate of the test efforts.

Scope

The levels of testing which will be applied are functional testing, unit testing and security testing. Parts of this project which won’t be covered by tests are the hardware parts, e.g. Raspberry Pi (Wifi, motion detection) and Arduino (LEDs, sound detection).

Intended Audience

This document is intended for people who are interested in the technical details of the project CLOUD. It explains the different types of testing which ensure the quality and reliability of the final product. This documentation is part of the project management and only interesting for people who will estimate this project.

  1. Evaluation Mission and Test Motivation

 The motivation and mission of this document is to improve the quality of software and to ensure a certain level of security.

Background

The security of the whole application has to be guaranteed. Errors that could crash the database, the Raspberry Pi, the Arduino or even the whole system have to be prohibited. This is very important as the final user / the customer has no access to the system itself and therefore cannot repair any problems. Usually, the customer doesn’t have the knowledge to repair the system or the web application when it crashes. The web application should therefore offer buttons

Evaluation Mission

The mission for the evaluation effort in this iteration is:

  • Ensure that every method / class works properly.
  • Ensure that the connection between backend and frontend is correct.
  • Ensure the integrity of the database.
  • Ensure the correctness of each database operation.
  • Satisfy the stakeholders / customers.
  • Ensure that the system has no crashes or errors.
  • Ensure the security of the Linux system.

Test Motivators

Our project contains a big amount of elements to be tested. The two big divisions are hardware and software. The first aspect relies on 100% functionality of the software which needs to be tested. We need to guarantee a good quality and smooth functionality of the software part to ensure the full functionality of the hardware. This project is sensitive for small dysfunctions, since the problem can’t be clearly found in two big divisions. Therefore we implemented many testing tools for back and front end.

  1. Testing Techniques and Types

Data and Database Integrity Testing

The database as itself and the database management system, which is in this case H2, won’t be tested directly. The H2 database engine is a open source JDBC api which has been already tested sufficiently. Therefore, no further research is necessary. The makers of H2 claim that their database is well tested using automated test cases.
Source: http://www.h2database.com/html/faq.html?highlight=tested&search=test#firstFound

  1. Function Testing

Function testing focuses on the GUI of the web application and ensures that each action on the GUIleads to the correct action in the backend.

Technique Objective:

  • Navigation through whole application
  • Visibility of all elements (menus, size, position, state, and focus)
  • Approve correct data transmission, processing and storing of data

Technique:

For each use case exists one feature file which tests the whole functionality of this use case. Each feature verifies that:

  • the expected results occur when valid data is used
  • the appropriate error or warning messages are displayed when         invalid data is used

Required Tools:

  • Cucumber
  • Selenium WebDriver
  • Ruby

Success Criteria:

All tests pass

  1. Business Cycle Testing

Business Cycle Testing is covered by the functional tests.

The following actions are tested:

  • Daily actions:
  • creating an event
  • uploading songs
  • creating playlists
  • Date-sensitive events:
  • creating an event
  • login
  • Actions that are performed every few seconds:
  • voting

Technique:

  • All time or date-sensitive functions will be executed using valid and invalid dates or time periods.
  • Testing will include using valid and invalid data to verify the following:
  • The expected results occur when valid data is used.
  • The appropriate error or warning messages are displayed when invalid data is used.

Required Tools:

  • Cucumber
  • Selenium WebDriver
  • Ruby

Success Criteria:

All tests passed

Special Considerations:

The time on the test system has to be up to date.

User Interface Testing

UI testing is covered by functional testing, here Cucumber.

Technique Objective:

  • Navigation through whole application
  • Visibility of all elements (menus, size, position, state, and focus)

Technique:

Selenium WebDriver

Oracles:

[Outline one or more strategies that can be used by the technique to accurately observe the outcomes of the test. The oracle combines elements of both the method by which the observation can be made and the characteristics of specific outcome that indicate probable success or failure. Oracles will be self-verifying, allowing automated tests to make an initial assessment of test pass or failure, however, be careful to mitigate the risks inherent in automated results determination.

Required Tools:

  • Cucumber
  • Selenium WebDriver
  • Ruby

Success Criteria:

The technique supports the testing of each screen or window that exists..

Special Considerations:

Some actions can be performed but not approved as they change parts of the hardware, e.g. lightning of Arduino. Those actions need to be tested directly by a human.

Security and Access Control Testing

Concerning security can be done a lot of testing.

Technique Objective:

  • Application-level Security:  an actor can access only those functions or data for which their user type is provided permissions.
  • System-level Security:  only those actors with access to the system and applications are permitted to access them.

Technique:

Parts that can and will  be tested:

  • Accessibility of dashboard
  • login
  • voting

Parts that cannot be tested:

  • access to database
  • opened ports
  • password for linux OS
  • general security of linux OS

Required Tools:

  • Cucumber
  • Selenium WebDriver
  • Ruby

Success Criteria:

All tests passed successfully

  1. Entry and Exit Criteria

Test Plan

Test Plan Entry Criteria

The functionality of the testing unit is given to >50% and requires test.

Test Plan Exit Criteria

 The functionality of the testing unit is given to >95% and tests were created (and passed).

 Suspension and Resumption Criteria

The testing unit is causing an unknown error which demands special treatment.

Test Cycles

Test Cycle Entry Criteria

The functionality of the testing unit is given to >80% and requires test.

Test Cycle Exit Criteria

The functionality of the testing unit is given to >99% and the tests reveal full functionality.

Test Cycle Abnormal Termination

The functionality of the testing unit is unknown and the issue is causing and internal or unknown error. New test cycle has to be created.

  1. Risks, Dependencies, Assumptions, and Constraints

Risk

Mitigation Strategy

Contingency (Risk is realized)

Prerequisite entry criteria is not met.

Tester will define the prerequisites that must be met before Load Testing can start.

Customer will endeavor to meet prerequisites indicated by Tester.

  • Meet outstanding prerequisites
  • Consider Load Test Failure

Test data proves to be inadequate.

Customer will ensure a full set of suitable and protected test data is available.

Tester will indicate what is required and will verify the suitability of test data.

  • Redefine test data
  • Review Test Plan and modify
  • components (that is, scripts)
  • Consider Load Test Failure

Database requires refresh.

System Admin will endeavor to ensure the Database is regularly refreshed as required by Tester..

  • Restore data and restart
  • Clear Database

The test strategies will be changed

Tester will learn more about current strategies and try to improve them.

  • Tester will have to adjust the new criteria and modify old files.

Test software will be changed

Tester has to update current software and take tutorials for better implementation.

  • Tester will have to adjust the new criteria and modify old files.

Problem Reporting, Escalation, and Issue Resolution

Reporting problems:

  • Comment on posts at Blog.licua.de
  • Create issue at Jira for project “CLOUD”
  • Create an issue at GitHub for project “CLOUD”

Hardware problem:

  1. Test to make sure the item isn’t really working correctly
  2. Try to find out the exact issue
  3. Ask google for proposal for solution
  4. Ask fellow students for proposal for solution
  5. Contact seller for information
  6. Send item back and replace it
  7. Test functionality
  8. Go to 1..

Software problem:

  1. Isolate issue
  2. Implement comments
  3. Test efficient the issue
  4. Ask google for proposal for solution
  5. Ask fellow students for proposal for solution
  6. Ask lecturer or professor for help
  7. If still no solution, go to 1., else:
  8. Implement solution
  9. Test it
  10. If still buggy, got to 1..
